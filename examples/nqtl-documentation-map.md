# Example: NQTL Documentation Map

Fictional and sanitized portfolio example. Not legal, compliance, or operational advice.

## Scenario

A health plan team needs to document how an NQTL is designed and applied across medical/surgical and mental health/substance use disorder benefits.

Sample NQTL: prior authorization for intensive outpatient services.

The team has several source documents, but the logic is spread across plan language, clinical criteria, operational workflows, committee oversight materials, and reporting outputs.

## Goal

Create a documentation map that helps reviewers understand:

- What the NQTL is
- Which benefits it applies to
- Which factors are used in design and application
- Which evidentiary standards support those factors
- How medical/surgical and MH/SUD processes compare as written
- What operational data would be needed to assess in-operation comparability
- Where the current documentation is strong, unclear, or missing

## Working Structure

| Step | Question | Output |
| --- | --- | --- |
| 1. Define the NQTL | What process, rule, or limitation is being reviewed? | Plain-language description and plan-term references |
| 2. Identify benefit classes | Which medical/surgical and MH/SUD benefits are in scope? | Benefit-class table |
| 3. List source documents | What policies, guidelines, criteria, workflows, and committee materials govern the process? | Source inventory |
| 4. Identify factors | What reasons or conditions cause the NQTL to apply? | Factor table |
| 5. Identify evidentiary standards | What evidence supports each factor? | Standard/source crosswalk |
| 6. Compare as written | Are the written processes comparable and no more restrictive for MH/SUD? | Narrative plus side-by-side table |
| 7. Compare in operation | What data shows how the process actually functions? | Data request and analysis plan |
| 8. Summarize findings | What is clear, what is missing, and what needs remediation? | Executive summary and action list |

## Sample Factor Table

| Factor | M/S application | MH/SUD application | Evidence to verify |
| --- | --- | --- | --- |
| Clinical appropriateness | Request reviewed against applicable medical criteria | Request reviewed against applicable behavioral health criteria | Criteria source, review workflow, clinical reviewer qualifications |
| Regulatory requirements | State/federal requirements incorporated into review timelines | State/federal requirements incorporated into review timelines | Regulatory matrix, policy references, timeliness reporting |
| Request for covered level of care | PA applies when plan terms require review for the service | PA applies when plan terms require review for the service | Plan documents, benefit grids, authorization lists |

## Questions I Would Ask

- Are the factor definitions written consistently across business areas?
- Are factors being used as design factors, application factors, or both?
- Can the team trace each factor to a source document?
- Are any process differences more restrictive for MH/SUD?
- If a process difference exists, is there a clear explanation and supporting evidence?
- What operational data is available, and does it match the written process?
- Are data limitations documented honestly?

## Example Data Request

- Authorization request counts by benefit type and service category
- Approval, denial, cancellation, and withdrawal rates
- Average time from complete request to decision
- Reviewer level involved in decision-making
- Peer review or escalation frequency
- Overturn rates, if applicable
- Audit findings related to criteria application or documentation quality

## Reviewer-Friendly Summary Format

1. What we reviewed
2. Why the NQTL exists
3. What factors and evidentiary standards are used
4. How M/S and MH/SUD compare as written
5. What the operational data shows
6. What limitations remain
7. What changes, if any, are recommended

## Why this example matters

Parity documentation can become overwhelming quickly. My approach is to make the logic visible: source by source, factor by factor, and workflow by workflow, so legal, clinical, compliance, and operations teams can see the same picture.
